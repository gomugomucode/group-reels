## Proposed Changes

We will perform the database updates via Supabase migrations, create new server functions, add query hooks, and modify the dashboard and group details pages.

### 1. Database Layer (Supabase Migrations)

#### [NEW] [20260713020000_group_collaboration.sql](file:///c:/Users/USER/Desktop/group-reels/supabase/migrations/20260713020000_group_collaboration.sql)
Creates the membership structure and updates video authorship mapping:
- Create group_members table with columns:
  - id UUID Primary Key default gen_random_uuid()
  - group_id UUID not null references public.groups(id) ON DELETE CASCADE
  - user_id UUID references auth.users(id) ON DELETE CASCADE (nullable to allow inviting unregistered emails)
  - email TEXT not null
  - role TEXT not null CHECK (role IN ('owner', 'member'))
  - invited_by UUID references auth.users(id) ON DELETE SET NULL
  - invitation_status TEXT not null CHECK (invitation_status IN ('pending', 'accepted', 'rejected')) default 'pending'
  - joined_at TIMESTAMPTZ (nullable)
  - created_at / updated_at TIMESTAMPTZ default now()
- Add UNIQUE constraint on group_members(group_id, email).
- Add index on group_members(group_id), group_members(user_id), and group_members(email).
- Add created_by column to video_links table (references auth.users(id) ON DELETE CASCADE with default value auth.uid()).
- Migration data population:
  - Backfill owner row in group_members for all existing groups.
  - Backfill created_by in video_links to match the owner of their group.
- Create automated on_group_created database trigger to insert the creator as the 'owner' in group_members upon group creation.
- Add Row Level Security (RLS) policies for group_members and update policies for groups and video_links to support collaborative roles.

### 2. Backend API / Server Functions

#### [NEW] [group-collaboration.functions.ts](file:///c:/Users/USER/Desktop/group-reels/src/lib/group-collaboration.functions.ts)
Contains TanStack Start server functions executing with admin privileges (to bypass RLS safely when checking invites or deleting records):
- inviteMember: Validates ownership, checks if email is already invited/member, and inserts a pending group_members record. If the email has a registered profile, pre-populates user_id.
- cancelInvitation / resendInvitation: Allows owners to remove pending invitations or update timestamps.
- acceptInvitation / rejectInvitation: Allows users matching the invited email to join or dismiss the team invitation.
- removeMember: Allows owner to delete a joined member's record.
- transferOwnership: Updates groups.created_by to the new owner's ID, demotes the old owner to 'member' in group_members, and promotes the new owner to 'owner'.
